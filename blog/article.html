v0.7：多进程 fork 服务器的三个 Bug

问题 1：控制台丢失请求详情

现象：启动 tcp_server 后控制台只输出 [#1] [#2] [#3]，丢失了请求方法和状态码。

根因：重构时将 handle_one_request() 替换为共享函数 request_handler_handle_connection()，但共享函数只写了 log_info（日志文件），没有写 printf（控制台）。

修复：在 send() 之前增加 printf("%s %s -> %d\n", method, path, resp_status);

教训：重构提取公共函数时，注意保留两种调用场景都需要的行为（日志 + 控制台输出）。

问题 2：bind 失败无终端提示 + 退出码 255

现象：第二个 fork server 实例启动时静默失败，退出码 255（-1 的无符号表示），终端无任何报错。

根因：两条链路叠加——fork server 缺少 SIGINT 处理器导致旧进程无法终止（见问题 3），端口占用；bind 失败时只写 log_error，没有 fprintf(stderr)。

修复：bind 失败分支增加 stderr 输出：ERROR: bind(127.0.0.1:8080) failed — port may already be in use

教训：系统调用失败时，stderr 输出比日志文件更直接——用户不会第一时间去看日志。

问题 3：Ctrl+C 无法终止 fork 服务器

现象：前台/后台运行 fork server 均无法用 Ctrl+C 优雅终止，旧进程一直占用端口。

根因：tcp_fork_server.c 只注册了 SIGCHLD（回收子进程）和 SIGPIPE（忽略），完全没有 SIGINT 相关代码。SIGINT 走默认行为直接杀进程，不执行清理代码。

修复：增加 g_shutdown 标志 + sigint_handler + sigaction 注册 + accept 循环检查。EINTR 分支中检查 g_shutdown 决定退出。

教训：网络服务器必须处理 SIGINT 实现优雅关闭，前台/后台运行都需要。测试脚本开头应清理残留进程（pkill）。

v0.10：wrk 吞吐量仅 ~1 QPS

问题现象

使用 wrk 对三种模式（fork/pool/select）压测时吞吐量仅 ~1 QPS，而 ab 和 http_bench.py 可达 18,000+ QPS。

wrk -t2 -c10 -d10s -R1000 --latency http://127.0.0.1:8080/hello
# → Requests/sec: ~1   ← 异常！

ab -c 10 -t 10 http://127.0.0.1:8080/hello
# → Requests per second: ~18000   ← 正常

定位过程

  1. 阅读 wrk 文档，确认 wrk 默认使用 HTTP/1.1 keep-alive 连接复用：一个 TCP 连接上连续发送多个请求。
  2. 追踪三种服务器模式的连接处理流程——全部是"一个连接 = 一个请求 + 一个响应"后立即关闭。
  3. request_handler_handle_connection() 内部只做一次 recv → 处理 → send 后直接 return，没有任何循环。
  4. 服务器响应虽标明 HTTP/1.1，但未包含 Connection: keep-alive 头，且响应发送完成后立即关闭连接。
  5. wrk 在第一次请求后尝试复用连接发送第二个请求，但服务器已关闭连接 → RST 或超时 → 必须重新 TCP 握手 → QPS 降到 ~1。

根因

服务器的连接处理模型是 HTTP/1.0 短连接（一个连接一个请求），wrk 的工作模型是 HTTP/1.1 keep-alive（连接复用）。两种模型冲突导致 wrk 大部分时间在等待重建连接。

修改内容（共 3 个文件）

文件修改
include/request_handler.h新增 KEEP_ALIVE_TIMEOUT_SEC=5、MAX_KEEP_ALIVE_REQUESTS=100
src/request_handler.cbuild_http_response() 添加 Connection: keep-alive；handle_connection() 添加 while 循环 + SO_RCVTIMEO 超时检测
src/select_server.cconn_t 新增 request_count/last_activity；select 超时 1s 周期性清理空闲连接

验证结果

修复后 wrk 吞吐量从 ~1 QPS 提升到 ~166 QPS（165× 提升），三种模式均通过。

v1.1：Keep-Alive 三轮排错

第一轮：AB Keep-Alive requests 始终为 0

现象：ab -k -c 50 -n 10000 /hello 测试，Keep-Alive requests: 0，吞吐量 1529 req/s（甚至比不用 -k 的 1666 req/s 更低）。

根因：服务器响应头中硬编码了 Connection: keep-alive（全小写），但 AB 2.3 对 Connection 头值做大小写敏感匹配，只认首字母大写的 Keep-Alive。AB 不认服务器的 keep-alive → 不复用连接。

修复：将所有 "keep-alive" 改为 "Keep-Alive"（request_handler.c、http_parser.c、epoll_server.c）。

第二轮：修复后仍为 0

现象：修改响应头后，ab -k 仍然 Keep-Alive requests: 0。

根因：抓包发现 ab -k 发送的是 HTTP/1.0 + Connection: Keep-Alive。服务器端 keep-alive 检测代码只匹配全小写 keep-alive——双方互相不认对方的 Connection 头！

修复：添加 HAS_KEEPALIVE / HAS_CLOSE 宏，覆盖 4 种大小写组合：Connection: keep-alive、Connection: Keep-Alive、connection: keep-alive、connection: Keep-Alive，以及对应的 close 变体。

第三轮：Connection reset by peer

现象：修复后 AB 中途崩溃——apr_socket_recv: Connection reset by peer (104)，5736 个请求后中止（目标 10000）。服务器端大量 "client disconnected (error)" 日志。

定位：5736 ≈ 50 并发 × 114 请求/连接，接近 MAX_KEEP_ALIVE_REQUESTS=100。服务器达到限额后调用 close() 时，AB 已预发送第 101 个请求在缓冲区中未读 → 内核发送 RST 而非 FIN。

修复：

  1. MAX_KEEP_ALIVE_REQUESTS 从 100 提升到 10000
  2. 实现 safe_close() 替代裸 close()：
    shutdown(fd, SHUT_RDWR) → drain 排空缓冲区 → close(fd)

shutdown(SHUT_RDWR) 通知对端"我不再收发数据"，触发正常的 TCP 四次挥手，不会产生 RST。

最终验证

模式QPSKeep-Alive 请求失败
Keep-Alive ON2,543 req/s10,000（100% 复用）0
Keep-Alive OFF764 req/s—0

性能提升：3.3×

v1.2.1：虚拟主机——通用静态文件路由缺失导致 CSS/JS 404

问题 1：CSS/JS/图片全部 404

现象：浏览器访问 blog.localhost 时，主页 HTML 正常但页面无样式——所有 CSS/JS/图片请求返回 404。curl 验证：/css/style.css → 404 NOT FOUND。

根因：服务器只有两个静态文件路由——GET /（提供 index.html）和 GET /blog/*（提供 blog 目录文件）。对于新站点 blog/ 和 lab/ 中 HTML 使用的相对路径（如 css/style.css → URI 为 /css/style.css），没有任何路由能匹配，全部落入兜底 404。

修复：在 /blog 路由和 API 路由之间新增通用静态文件处理器——对任意 GET/HEAD 路径，从当前虚拟主机 root 提供文件，同时保留路径穿越安全检查（403 Forbidden）。

关键代码：http_serve_file(g_current_root, req->path, ...)——g_current_root 由 Host 头匹配出的 server block 动态确定（"blog" / "lab" / "www"）。

教训：静态文件服务器必须支持从根目录提供任意路径的文件，不能只覆盖特定的 URL 前缀。HTML 的相对路径引用不会自动带上虚拟主机前缀。

问题 2：blog.local 不解析 → 502

现象：浏览器访问 http://blog.local:8080/ 返回 502 Bad Gateway。

根因：blog.local 不是特殊域名，未被 DNS 或 /etc/hosts 解析到 127.0.0.1，请求发到了外网。

修复：每个 server block 的 server_name 同时添加 *.localhost 变体：server_name blog.local blog.localhost。*.localhost 在所有系统上自动指向 127.0.0.1，无需任何配置即可测试虚拟主机。

教训：测试虚拟主机时 *.localhost 是最佳域名选择——零配置开箱即用。

问题 3：端口 80 需 root 权限

现象：启动时 bind(0.0.0.0:80) failed，退出码 255。

根因：1024 以下端口是 Linux 特权端口，普通用户无权绑定。

修复：默认配置端口从 80 改为 8080。

经验总结

  1. HTTP 头值大小写不敏感是规范要求，但现实中的工具（如 AB）不一定遵守。服务器应兼容所有大小写变体——不要只测 curl，也要测 ab 和 wrk。
  2. AB 的 -k 标志发送的是 HTTP/1.0 + Connection: Keep-Alive，而非 HTTP/1.1。设计 keep-alive 协商逻辑时必须考虑 HTTP/1.0 显式 keep-alive 的情况。
  3. 服务器主动关闭 keep-alive 连接时必须用 shutdown() + drain + close() 三步走，否则客户端预发送的数据会导致 RST 而非正常挥手。Benchmark 工具的 Connection reset by peer 错误几乎总是因为这个。
  4. "长连接"在中文中可指"请求处理时间长"或"HTTP keep-alive 连接复用"，二者是完全不同的概念，需在文档中明确区分。
  5. SO_RCVTIMEO 是实现 keep-alive 空闲超时的简单有效方式（fork/pool 模式）；select() 超时参数可用于周期性任务（如空闲连接清理），是 select 模式下的自然选择。
  6. 重构提取公共函数时，注意保留两种调用场景都需要的行为（日志 + 控制台输出）。
  7. 系统调用失败时 stderr 输出比日志更直接——用户不会第一时间去看日志文件。
  8. 网络服务器必须处理 SIGINT 实现优雅关闭。
  9. 测试脚本开头应清理残留进程（pkill），避免上次运行的进程干扰本次测试。
