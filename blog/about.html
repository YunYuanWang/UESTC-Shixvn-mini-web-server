整体架构（v1.2 Master-Worker 模式）

                    ┌──────────────────────────────────┐
                    │            conf/server.conf         │
                    │  host, port, worker_processes, ...  │
                    └──────────────┬───────────────────┘
                                   │ load_config()
                    ┌──────────────▼───────────────────┐
                    │           Master Process          │
                    │  ┌─────────────────────────────┐  │
                    │  │ socket() → bind() → listen() │  │
                    │  │ user_store_load_csv()        │  │
                    │  │ fork() × worker_processes    │  │
                    │  │ SIGINT → graceful shutdown   │  │
                    │  └─────────────────────────────┘  │
                    └──────┬──────────┬──────────┬──────┘
                           │ fork()   │ fork()   │
                    ┌──────▼───┐ ┌───▼──────┐ ┌▼──────────┐
                    │ Worker-1 │ │ Worker-2 │ │ Worker-N   │
                    │ log_reopen()         │ │ ...        │
                    │ epoll event loop     │ │            │
                    │ accept → recv → 解析  │ │            │
                    │ → 路由 → send → log  │ │            │
                    │ SIGTERM → safe_close │ │            │
                    └──────────┘ └──────────┘ └────────────┘

Master 进程职责

  • 配置加载：解析 conf/server.conf，支持两种格式——传统 flat key=value 和 Nginx 风格 server { } 块
  • 虚拟主机：解析 server_name 指令，建立域名→文档根目录的映射表
  • 数据加载：调用 user_store_load_csv() 一次，fork 后各 Worker 通过 CoW 共享
  • 监听 Socket：socket() → setsockopt(SO_REUSEADDR) → bind() → listen()
  • Worker 管理：fork N 个子进程，记录 PID
  • 信号处理：SIGINT 触发优雅关闭，SIGCHLD 回收已退出 Worker
  • 优雅关闭：关闭 listen socket → SIGTERM 通知 Worker → waitpid 超时等待 → SIGKILL 兜底

Worker 进程职责

  • 独立 Epoll 事件循环：每个 Worker 在继承的 listen_fd 上独立运行 epoll_wait/accept
  • HTTP 请求处理：recv → 解析请求行/头/体 → 路由分发 → 构建响应 → send
  • Keep-Alive 管理：HTTP 版本协商 + Connection 头检测 + 空闲超时 + 最大请求限制
  • 独立日志：fork 后调用 log_reopen() 获取独立 FILE*，O_APPEND 保证原子写入
  • 优雅退出：收到 SIGTERM 后设 g_shutdown 标志，完成当前请求后 _exit(0)

Epoll 事件驱动核心

epoll_create1(0)                        // 创建 epoll 实例
epoll_ctl(ADD, listen_fd, EPOLLIN)      // 注册监听 socket

while (!g_shutdown) {
    nfds = epoll_wait(epfd, events, MAX, 1000);  // 1s 超时

    for each ready event:
        if fd == listen_fd:
            conn_fd = accept()
            epoll_ctl(ADD, conn_fd, EPOLLIN)      // 注册客户端
            conn->last_activity = time(NULL)

        else:  // client data
            n = recv(fd, buf, ...)
            if n <= 0 → safe_close(fd)
            else:
                解析 HTTP → request_handler_process_http()
                → send(fd, resp_buf, resp_len)
                → keep-alive 或 close

    // 超时扫描：清理 idle > KEEP_ALIVE_TIMEOUT_SEC 的连接
}

关键技术点：

  • Level-Triggered (LT)：默认水平触发模式，与 select 语义一致，简单可靠
  • conn_t 数组：固定大小（EPOLL_MAX_CONNS=1024），epoll_event.data.ptr 直接指向 conn_t，O(1) 查找
  • safe_close()：shutdown(SHUT_RDWR) → drain → close()，避免 TCP RST
  • 超时扫描：利用 epoll_wait 的 1s timeout 周期性检查空闲连接

HTTP 请求处理流程

Client → recv() → 解析请求行 (METHOD /uri HTTP/1.x)
                   → 解析头部 (Content-Length, Connection, ...)
                   → 解析 Body (基于 Content-Length)
                   → Keep-Alive 协商 (HTTP 版本 + Connection 头)
                   → 路由分发 (request_handler_process_http)

路由表:
  GET  /              → {root}/index.html (text/html)
  GET  /blog/         → blog/index.html (博客首页)
  GET  /blog/*        → blog/ 静态文件 (动态 MIME)
  GET  /*             → {root}/ 通用静态文件 (v1.2.1 新增)
  GET  /hello         → "Hello, Web!\n"
  GET  /users         → 用户列表 (BST 中序遍历)
  GET  /users/  → 查找用户
  POST /users         → 添加用户 (CSV body)
  DELETE /users/ → 删除用户
  ...
  {root} = 由 Host 头匹配的 server block 确定

名称虚拟主机（v1.2.1 新增）

参考 Nginx 的 server { } 块语法，通过 HTTP Host 头将不同域名路由到不同文档根目录，实现一个服务器实例托管多个独立网站。

配置格式

worker_processes 2;

server {
    listen 0.0.0.0:8080 default_server;
    server_name localhost;
    root www;
}

server {
    listen 0.0.0.0:8080;
    server_name blog.local blog.localhost;
    root blog;
}

server {
    listen 0.0.0.0:8080;
    server_name lab.local lab.localhost;
    root lab;
}

请求路由流程

1. recv() 读取 HTTP 请求原始字节
2. 扫描 "\nHost: " 头值（大小写不敏感）
3. 提取 hostname，剥离端口后缀
4. config_find_server(hostname):
   → 遍历 server blocks 的 server_names[]
   → 大小写不敏感精确匹配
   → 未匹配则使用 default_server
5. request_handler_set_root(block->root)
6. 后续处理与原单网站模式完全相同

数据流

curl -H "Host: blog.localhost" http://127.0.0.1:8080/
  → Host 头 "blog.localhost" → 匹配 blog 块 → root="blog"
  → http_serve_file("blog", "/", ...) → blog/index.html

curl -H "Host: lab.localhost" http://127.0.0.1:8080/
  → Host 头 "lab.localhost" → 匹配 lab 块 → root="lab"
  → http_serve_file("lab", "/", ...) → lab/index.html

curl http://127.0.0.1:8080/
  → 无 Host 头 → default_server → root="www"
  → http_serve_file("www", "/", ...) → www/index.html

日志系统（v1.2）

系统日志与访问日志分离，支持四级日志和行数阈值自动滚动：

// API 示例
log_infof("Worker-%d PID: %d", id, pid);       // printf 风格
log_error("accept() failed");                   // 简单字符串
log_access(client_ip, method, path, 200, len);  // 访问记录

// 输出格式
// system.log:  [2026-07-19 15:30:45.123456] [INFO ] [PID 12345] ... message
// access.log:  [19/Jul/2026:15:30:45 +0800] 127.0.0.1 GET /blog/ 200 4582

// 滚动机制: log_max_lines=10000, log_max_roll_files=5
// server.log → server.log.1 → server.log.2 → ... → remove
