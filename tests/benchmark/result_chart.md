## 三、三种模型的性能边界

### 1. Fork（多进程）

**性能边界：单连接 ~50 QPS，并发 >1 即崩溃**

| 瓶颈 | 表现 | 原因 |
|------|------|------|
| **并发上限** | **c=2 即拒绝连接** | `accept()` 队列被快速填满。fork() 调用约 0.1-0.5ms，加上 SIGCHLD 处理 + waitpid()，单线程 accept 循环无法跟上并发连接速率 |
| **吞吐上限** | **~50 QPS**（单连接） | 每个请求：accept() → fork() → 子进程 recv/send/exit → 父进程 close(fd) + SIGCHLD。fork() 系统调用是核心瓶颈 |
| **内存边界** | 每连接 +2.4MB（进程 RSS） | 每个子进程继承父进程地址空间（CoW），100K 用户数据集下每进程 ~2.4MB。100 并发 = 240MB+ |
| **CPU 特征** | sys% 远高于 user%（内核调度开销） | 进程创建/销毁/调度的上下文切换消耗大量 CPU 时间 |
| **长请求表现** | 理论上真并行，实际崩溃 | CPU 密集型请求下每个进程独立计算，可多核并行。但在本 VM 上连接数稍多即崩溃 |
| **资源耗尽类型** | 进程数超限 / 内存 OOM / PID 耗尽 | `ulimit -u` 限制最大进程数 |

> **适用场景**：需要进程隔离的安全敏感场景，并发 < 5 的后台管理接口。不适合任何高并发场景。

### 2. Pool（线程池）

**性能边界：~10,000 QPS 吞吐天花板，P95 延迟随并发线性增长**

| 瓶颈 | 表现 | 原因 |
|------|------|------|
| **吞吐天花板** | **~10,000 QPS**（各并发级别均在此附近） | 线程池最大 8 个 worker（`MAX_POOL_SIZE=8`）。当 8 个 worker 全部饱和时，新请求在队列中等待。队列容量 128（`QUEUE_CAPACITY`），队列满后 `thread_pool_add_task()` 阻塞主线程的 accept() |
| **延迟拐点** | **c>50 时 P95 突破 10ms** | 线程竞争 + 队列排队。worker 数固定（8），请求超过 worker 处理能力时排队延迟主导 |
| **CPU 极限** | **~140% CPU**（多核利用） | 8 个 worker 线程可占用多个核心。CPU 利用率超过 100% 说明多核并行有效 |
| **内存** | **~18MB RSS**，稳定 | 线程共享地址空间，额外开销仅为线程栈（每线程 ~8MB 虚拟内存，实际 RSS 很小） |
| **长请求表现** | **可处理但排队严重** | 每个 `/users/compare` 耗时 30-45s，8 个 worker 同时处理 8 个请求。第 9 个请求在队列中等待 ~30s。c=10 时平均延迟 24.6s（因排队） |

> **适用场景**：通用 Web 服务，CPU 密集型 + I/O 密集型混合负载。可通过增大 `MAX_POOL_SIZE` 提升吞吐天花板。

### 3. Select（I/O 多路复用）

**性能边界：~12,000 QPS，FD_SETSIZE=1024 为硬限制，CPU 密集型请求阻塞整个事件循环**

| 瓶颈 | 表现 | 原因 |
|------|------|------|
| **吞吐天花板** | **~12,000 QPS**（接近单核理论上限） | 单线程事件循环，`select()` 系统调用 + O(n) fd 扫描是主要开销。每请求 ~0.08ms（12K QPS），已接近单核处理极限 |
| **延迟特征** | **随并发线性增长** | 单线程串行处理所有请求。c=1100 时 P95=120ms（所有请求排队等待事件循环处理） |
| **FD_SETSIZE 硬限制** | **同时打开 >1024 个 fd 时拒绝连接** | `SELECT_MAX_CONNS=1024`，`FD_SETSIZE` 通常为 1024。本测试中因连接为短连接（处理完即关闭），实际未触发此限制。若使用 keep-alive 长连接，c>1024 即报 "connection table full" |
| **CPU 特征** | **~100% CPU（单核打满）** | 无锁无切换，CPU 全部用于业务处理。user% 高，sys% 低。这是最优的 CPU 利用模式 |
| **内存** | **~7-20MB RSS**，最轻量 | 单进程单线程，无额外内存开销 |
| **长请求（致命缺陷）** | **c=5 即全部超时，QPS=0** | **单线程阻塞**：一个 `/users/compare` 计算 30 秒，期间**所有其他连接完全阻塞**。这是 select 模式的阿克琉斯之踵。在此场景下 select 完全不可用 |

> **适用场景**：大量短连接（静态文件、API 网关）、I/O 密集型（代理转发）。**严禁**用于有 CPU 密集型运算的服务。

### 选型建议

1. **通用 Web 服务** → **Pool（线程池）**：均衡的吞吐/延迟/稳定性，调整 `MAX_POOL_SIZE` 可适配不同负载
2. **API 网关 / 静态文件 / 代理** → **Select（I/O 复用）**：短请求场景最优，无锁无切换。但必须升级为 epoll 避免 FD_SETSIZE 限制和 O(n) 扫描
3. **需要进程隔离 / 安全第一** → **Fork（多进程）**：极低并发（<5），如管理后台，每个请求独立进程防止互相影响
4. **混合场景（短请求 + 偶尔长请求）** → **epoll + 线程池**（需重构）：主线程 epoll 接收连接，worker 线程池处理 CPU 密集型请求。避免 select 的长请求阻塞问题
