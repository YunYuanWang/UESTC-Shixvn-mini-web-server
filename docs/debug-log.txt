问题 5: wrk2 压测吞吐量仅 ~1 QPS（服务器无 HTTP/1.1 keep-alive 支持）
================================================================

问题现象：
  使用 wrk2 对三种模式（fork/pool/select）进行并发压测时，吞吐量仅 ~1 QPS，
  而 ab 和 http_bench.py 可达 18,000+ QPS。wrk2 的测试结果与预期严重不符。

复现命令：
  make clean && make
  ./mini_web_server pool 127.0.0.1 8080 &
  wrk2 -t2 -c10 -d10s -R1000 --latency http://127.0.0.1:8080/hello
  → 结果: Requests/sec: ~1

  # 对比 ab（正常）:
  ab -c 10 -t 10 http://127.0.0.1:8080/hello
  → 结果: Requests per second: ~18000

定位过程：
  1. 阅读 wrk2 文档，确认 wrk2 默认使用 HTTP/1.1 keep-alive 连接复用：
     一个 TCP 连接上连续发送多个请求，等待全部响应后再发送下一批。
  2. 追踪三种服务器模式的连接处理流程：
     - Fork 模式 (tcp_fork_server.c:216-224):
       request_handler_handle_connection(conn_fd) → shutdown + close + _exit
     - Pool 模式 (thread_pool.c worker):
       request_handler_handle_connection(client_fd) → close(client_fd)
     - Select 模式 (select_server.c:451-461):
       send() 响应后立即 FD_CLR + close(fd) + conn->fd = -1
  3. request_handler_handle_connection() 内部只做一次 recv() → 处理 → send()
     后直接 return，没有任何循环。
  4. 服务器响应虽标明 HTTP/1.1，但未在响应中包含 Connection: keep-alive 头，
     且响应发送完成后立即关闭连接。
  5. wrk2 在第一次请求收到响应后，尝试在同一连接上发送第二个请求，
     但服务器已关闭连接 → wrk2 收到 RST 或连接超时 → 必须重新 TCP 握手。

原因分析：
  服务器的连接处理模型是"一个 TCP 连接 = 一个 HTTP 请求 + 一个响应"，
  这是 HTTP/1.0 时代的短连接模型。wrk2 的工作模型是 HTTP/1.1 keep-alive：
  一个 TCP 连接上复用多个请求/响应对，避免重复的 TCP 握手开销。

  两种模型的冲突导致 wrk2 的连接复用机制完全失效：每次请求都需要
  wrk2 重新建立 TCP 连接，而服务器的 close() 时序与 wrk2 的重用时机
  不匹配，导致 wrk2 大部分时间在等待重建连接，实际 QPS 降到 ~1。

  注：项目的"长连接测试"指的是请求处理时间长（如 /users/compare 单次
  5~30 秒），而非 HTTP 协议层面的 keep-alive 连接复用。二者概念不同。

修改内容：
  共修改 3 个文件，添加标准 HTTP/1.1 keep-alive 支持：

  1. include/request_handler.h:
     新增常量 KEEP_ALIVE_TIMEOUT_SEC=5（空闲超时）、
     MAX_KEEP_ALIVE_REQUESTS=100（单连接最大请求数）

  2. src/request_handler.c（fork/pool 模式共用）:
     - build_http_response(): 响应头添加 Connection: keep-alive
     - request_handler_handle_connection():
       设置 SO_RCVTIMEO 实现空闲超时检测
       添加 while 循环（max MAX_KEEP_ALIVE_REQUESTS 次迭代）
       每次迭代: recv() → 解析 → 处理 → send() → request_count++
       recv 超时（EAGAIN/EWOULDBLOCK）或客户端关闭（recv=0）时退出循环
       内联 400/500 错误响应添加 Connection: close

  3. src/select_server.c（select 模式）:
     - conn_t 新增 request_count 和 last_activity 字段
     - select() 超时从 NULL 改为 1 秒，用于周期性检查空闲连接
     - 超时分支: 遍历所有连接，关闭 idle > 5s 的连接
     - accept 时初始化 last_activity = time(NULL)
     - recv 时更新 last_activity
     - 成功响应后不再关闭连接，改为:
       重置 recv_buf/recv_len，fd 保留在 master_set 中等待下一次数据
       到达 MAX_KEEP_ALIVE_REQUESTS 后关闭连接

  tcp_fork_server.c / thread_pool.c / tcp_pool_server.c 无需修改：
  keep-alive 循环内置于 request_handler_handle_connection() 中，
  调用者仍然在 handler 返回后关闭 fd。

验证结果：
  make clean && make  → 编译无警告

  # curl keep-alive 验证（三种模式均通过）:
  curl -v --keepalive-time 5 http://127.0.0.1:8080/hello http://127.0.0.1:8080/hello
  → < Connection: keep-alive
  → * Connection #0 to host 127.0.0.1 left intact
  → * Reusing connection #0  ✓

  # wrk2 压测验证（三种模式均通过）:
  wrk2 -t2 -c10 -d5s -R500 --latency http://127.0.0.1:8080/hello
  → Fork:   1000 requests/6.02s, 166 req/sec, 0 read/write errors  ✓
  → Pool:   1000 requests/6.02s, 166 req/sec, 0 read/write errors  ✓
  → Select: 1000 requests/6.03s, 166 req/sec, 0 read/write errors  ✓

  修复前: ~1 QPS  →  修复后: ~166 QPS（165x 提升）

已知限制：
  HTTP pipelining（在一个 TCP segment 中同时发送多个请求）不支持。
  如 nc 一次性发送两个请求，服务器第一次 recv() 会读走全部数据，
  处理完第一个请求后，第二个请求的字节已丢失。wrk2 不受影响——
  它使用非流水线的 keep-alive（发一个请求，等完整响应，再发下一个）。

问题 2: fork 服务器退出码 255（bind 失败无终端提示）
================================================================

问题现象：
  $ ./mini_web_server fork & sleep 5
  [2] 274021
  [#4] child PID 274224 handling request
  GET /users -> 200
  [2]+  退出 255              ./mini_web_server fork

  服务器退出码 255（-1 的无符号表示），且 [#4] 从第 4 个开始计数。

复现命令：
  # 终端 1
  ./mini_web_server fork &
  # 终端 2（不等终端 1 的服务器退出）
  ./mini_web_server fork & sleep 5

定位过程：
  1. cat logs/server.log 查看日志，发现关键错误行：
     [ERROR] [PID 274021] bind(127.0.0.1:8080) failed
  2. 日志中有两个 fork server 实例：
     PID 272024 — 第一个，正常运行
     PID 274021 — 第二个，bind 失败
  3. 第二个实例 bind 失败 → tcp_fork_server_run() 返回 -1 → main() 返回 -1
     → bash 显示退出码 255（-1 & 0xFF = 255）
  4. [#4] 的输出来自第一个实例(PID 272024)，不是用户刚启动的第二个。
  5. bind 失败的原因：第一个 fork server 没有 SIGINT 处理器，
     Ctrl+C 无法终止后台进程，端口一直被占用。

原因分析：
  两条链：
  a) fork server 缺少 SIGINT 处理器（见问题 3），导致 Ctrl+C 无法终止，
     端口被旧进程占用。
  b) bind 失败时只写了 log_error（日志文件），没有 fprintf(stderr, ...)，
     用户在终端完全看不到失败原因。

修改内容：
  src/tcp_fork_server.c: bind 失败分支增加
    fprintf(stderr, "ERROR: bind(127.0.0.1:8080) failed — "
            "port may already be in use\n");

验证结果：
  make && ./mini_web_server fork & sleep 0.3 && ./mini_web_server fork
  → stderr 输出: ERROR: bind(127.0.0.1:8080) failed — port may already be in use
  → 退出码 255 的原因对用户可见 ✓

问题 3: Ctrl+C 无法终止 fork 服务器（缺少 SIGINT 处理器）
================================================================

问题现象：
  前台运行 ./mini_web_server fork，按 Ctrl+C 后进程不会优雅退出；
  后台运行 ./mini_web_server fork &，无法用常规方式终止。
  旧进程一直占用 8080 端口，导致新实例 bind 失败。

复现命令：
  ./mini_web_server fork &
  kill -INT $!        # SIGINT 无法终止
  kill $!             # SIGTERM 默认行为：终止，但不优雅

定位过程：
  1. 阅读 tcp_fork_server.c 信号设置代码，发现只有：
       sigaction(SIGCHLD, ...)   // 回收子进程
       signal(SIGPIPE, SIG_IGN)  // 忽略管道断裂
     完全没有 SIGINT 相关代码。
  2. 对比 tcp_server.c，后者有：
       static volatile sig_atomic_t g_shutdown = 0;
       static void sigint_handler(int sig) { g_shutdown = 1; }
       sigaction(SIGINT, &sa, NULL);
       while (!g_shutdown) { accept... }
  3. fork server 的 while 条件是 while (clients_served < MAX_CLIENTS)，
     没有响应 SIGINT 的机制。

原因分析：
  缺少三个部分：
  a) SIGINT 信号处理器（设 g_shutdown 标志）
  b) sigaction 注册 SIGINT
  c) accept 循环中检查 g_shutdown 决定是否退出
  导致 SIGINT 走默认行为：直接杀进程，不执行清理代码（close listen_fd、
  回收子进程等）。

修改内容：
  src/tcp_fork_server.c:
  1. 增加 g_shutdown 标志和 sigint_handler:
       static volatile sig_atomic_t g_shutdown = 0;
       static void sigint_handler(int sig) { (void)sig; g_shutdown = 1; }
  2. 信号设置中注册 SIGINT:
       sa.sa_handler = sigint_handler;
       sigaction(SIGINT, &sa, NULL);
  3. accept 循环条件改为 while (clients_served < MAX_CLIENTS && !g_shutdown)
  4. EINTR 分支中检查 g_shutdown:
       if (errno == EINTR) { if (g_shutdown) break; continue; }
  5. 退出日志中标记 SIGINT 退出:
       printf("ForkServer: served %d clients, exiting%s.\n",
              clients_served, g_shutdown ? " (Ctrl-C)" : "");

验证结果：
  ./mini_web_server fork &
  curl -s http://127.0.0.1:8080/hello > /dev/null
  kill -INT $!
  → 控制台输出: ForkServer: served 1 clients, exiting (Ctrl-C).
  → exit code = 0, 端口释放 ✓

问题 1：Keep-Alive requests 始终为 0，ab -k 性能反而更差
================================================================

问题现象：
  ab -k -c 50 -n 10000 http://127.0.0.1:8080/hello
  → Keep-Alive requests: 0
  → 吞吐量 1529 req/s（比不用 -k 的 1666 req/s 还低）

  不用 -k（HTTP/1.0，每次新建连接）:
  → 吞吐量 1666 req/s

  预期：-k 启用 keep-alive 后吞吐量应显著提升，Keep-Alive requests 应为 9950+。

定位过程：
  1. 检查服务器响应头，确认发送的是 Connection: keep-alive（全小写）。
  2. 阅读 AB 2.3 源码，发现 AB 对 Connection 头值做大小写敏感匹配：
     AB 期望 Connection: Keep-Alive（首字母大写），而服务器发送 keep-alive（全小写）。

    "Connection: keep-alive\r\n"
  AB 2.3 使用大小写敏感比较，只认 "Keep-Alive"（首字母大写）。
  这导致 AB 认为服务器不支持 keep-alive，所有请求仍走短连接模式。

修改内容：
  1. src/request_handler.c: "Connection: keep-alive" → "Connection: Keep-Alive"
  2. src/http_parser.c:  "keep-alive" → "Keep-Alive"
  3. src/epoll_server.c: strstr 匹配字符串同步更新

问题 2：修复问题 1 后，ab -k 仍为 Keep-Alive requests: 0
================================================================

问题现象：
  ab -k -c 50 -n 10000 http://127.0.0.1:8080/hello
  → Keep-Alive requests: 0
  → 吞吐量仍无提升

定位过程：
  1. 抓包分析 AB 发送的请求，发现 ab -k 实际发送的是：
     GET /hello HTTP/1.0\r\nConnection: Keep-Alive\r\n...
  2. AB 用 HTTP/1.0 + 显式 Connection: Keep-Alive 头来启用 keep-alive。
  3. 服务器端 keep-alive 检测代码:
     if (strstr(conn->recv_buf, "HTTP/1.0")) {
         close_after = 1;   // HTTP/1.0 默认关闭
         if (strstr(conn->recv_buf, "Connection: keep-alive") ||  // ← 全小写
             strstr(conn->recv_buf, "connection: keep-alive")) {  // ← 全小写
             close_after = 0;
         }
     }
  4. AB 发送 Connection: Keep-Alive（大写 K/A），代码只检查全小写 keep-alive。
  5. 大小写不匹配 → close_after 保持为 1 → req.keep_alive = 0 → 关闭连接。

  两处大小写不敏感问题叠加：

  b) 请求头 Connection: Keep-Alive（混合大小写）→ 服务器不认（问题 2）
  双方互相不认对方的 Connection 头，keep-alive 完全失效。

修改内容：
  src/epoll_server.c: 添加 HAS_KEEPALIVE / HAS_CLOSE 宏，覆盖 4 种大小写组合：
    "Connection: keep-alive" / "Connection: Keep-Alive"
    "connection: keep-alive" / "connection: Keep-Alive"
    以及对应的 close 变体。

问题 3：Connection reset by peer（修复问题 2 后 AB 中途崩溃）
================================================================

问题现象：
  ab -k -c 50 -n 10000 ...
  → apr_socket_recv: Connection reset by peer (104)
  → Total of 5736 requests completed（10000 未完成）

  服务器控制台输出大量:
  [-] client disconnected (error): 127.0.0.1:60328
  [-] client disconnected (error): 127.0.0.1:60316

定位过程：
  1. 5736 ≈ 50 并发 × 114 请求/连接，接近 MAX_KEEP_ALIVE_REQUESTS = 100。
  2. 推测：连接达到 100 请求上限后服务器调用 close()，但 AB 已经发送了
     第 101 个请求（HTTP/1.0 keep-alive pipelining 或激进的预发送）。
  3. 服务器 close() 时 socket 接收缓冲区有未读数据 → 内核发送 RST 而非 FIN。
  4. AB 收到 RST → apr_socket_recv 报 Connection reset by peer。

原因分析：
  TCP 协议规定：如果 close() 时接收缓冲区有未读数据，内核发送 RST 而非
  正常的 FIN 挥手序列。这是为了避免数据丢失——告诉对端"你发的数据我没读就丢了"。

  AB 在 keep-alive 模式下可能预发送下一个请求（不等上一个响应完全处理完），
  当服务器因达到 MAX_KEEP_ALIVE_REQUESTS 而 close() 时，预发送的数据还在
  缓冲区中未读 → RST。

修改内容：
  1. include/request_handler.h: MAX_KEEP_ALIVE_REQUESTS 100 → 10000
     （100 的限额对 ab -n 10000 压测不够，50 连接 × 100 = 5000 请求后就触发关闭）

  2. src/epoll_server.c: 新增 safe_close() 函数替代裸 close():
     static void safe_close(int fd) {
         shutdown(fd, SHUT_RDWR);    // 优雅关闭：先发 FIN
         char dummy[256];
         while (recv(fd, dummy, sizeof(dummy), 0) > 0) {}  // 排空缓冲区
         close(fd);
     }
     所有 close(ready_fd) 和 close(connections[i].fd) 替换为 safe_close()。

     shutdown(SHUT_RDWR) 的作用：通知对端"我不再收发数据"，触发正常的
     TCP 四次挥手（FIN → ACK → FIN → ACK），不会产生 RST。
     recv 排空循环确保缓冲区中没有残留数据。

验证结果：
  # Keep-Alive ON:
  ab -k -c 50 -n 10000 http://127.0.0.1:8080/hello
  → Keep-Alive requests: 10000   ✓ 全部复用连接
  → 吞吐量 2543 req/s
  → Time taken: 3.931s
  → Failed requests: 0

  # Keep-Alive OFF（对比）:
  ab -c 50 -n 10000 http://127.0.0.1:8080/hello
  → 吞吐量 764 req/s
  → Time taken: 13.085s

  性能提升: 2543 / 764 = 3.3x

v1.1 keep-alive 调试总结
================================================================

| 问题 | 根因 | 修复位置 |
|------|------|---------|
| AB Keep-Alive requests: 0 | 响应 Connection 头值为全小写 "keep-alive"，AB 大小写敏感，不认 | request_handler.c, http_parser.c, epoll_server.c |
| 修复后仍为 0 | AB 发 HTTP/1.0 + Connection: Keep-Alive，服务器只匹配全小写变体 | epoll_server.c (HAS_KEEPALIVE 宏) |
| Connection reset by peer | MAX_KEEP_ALIVE_REQUESTS=100 时 close() 触发 RST（缓冲区有未读数据） | request_handler.h (100→10000), epoll_server.c (safe_close) |

经验教训：
1. HTTP 头值大小写不敏感是规范要求，但现实中的工具（如 AB）不一定遵守。
   服务器应兼容所有大小写变体。
2. AB 的 -k 标志发送的是 HTTP/1.0 + Connection: Keep-Alive，而非 HTTP/1.1。
   设计 keep-alive 协商逻辑时必须考虑 HTTP/1.0 显式 keep-alive 的情况。
3. 服务器主动关闭 keep-alive 连接时，必须用 shutdown() + drain + close()
   三步走，否则客户端预发送的数据会导致 RST 而非正常挥手。
4. Benchmark 工具的 Connection reset by peer 错误几乎总是因为服务器端
   直接 close() 了一个有未读数据的 socket。

问题 1：通用静态文件路径全部 404（CSS/JS/图片加载失败）
================================================================

问题现象：
  浏览器访问 http://blog.localhost:8080/ 时，HTML 主页正确返回 200，
  但页面中引用的 css/style.css、js/main.js、img/logo.png 等资源全部
  返回 404，页面只有裸 HTML 无样式。

  用 curl 验证：
    curl -H "Host: blog.localhost" http://127.0.0.1:8080/css/style.css
    → HTTP/1.1 404 NOT FOUND

定位过程：
  1. 追踪 request_handler_process_http() 的路由分发逻辑：
     ① GET / → 精确匹配 "/"，调用 http_serve_file(g_current_root, ...)
     ② /blog* → 前缀匹配，调用 http_serve_file("blog", ...)
     ③ 其他 API 路由（/hello, /users, /sleep 等）
     ④ 兜底 → 404
  2. css/style.css 的 path 是 "/css/style.css"：
     - 不匹配 "/"（精确匹配失败）
     - 不匹配 "/blog"（前缀不匹配）
     - 不匹配任何 API 路由
     → 落入 404
  3. 原版只托管 www/ 和 blog/ 两个目录，所有资源都在 /blog/ 路径下，
     HTML 中直接用 <link href="/blog/css/style.css"> 这种绝对路径。
     但新站点 blog/ 和 lab/ 中 HTML 引用的是相对路径 css/style.css
     （即 /css/style.css），不在 /blog 前缀下。

原因分析：
  服务器只有两个静态文件路由：
  - GET /          → g_current_root/index.html
  - GET /blog/*    → blog/ 目录下的文件
  对于任意路径如 /css/style.css、/js/main.js、/img/logo.png，没有
  对应的路由处理器。这导致新站点（blog.localhost、lab.localhost）的
  所有非 HTML 资源全部 404。

修改内容：
  src/request_handler.c: request_handler_process_http()
  在 /blog 路由和 API 路由之间，新增通用静态文件处理器：
    if (strcmp(req->method, "GET") == 0 || strcmp(req->method, "HEAD") == 0) {
        // 安全检查：拒绝路径穿越（..、~、\）
        if (!http_is_safe_path(req->path)) → 403 Forbidden
        // 从当前虚拟主机的 root 提供任意文件
        int file_len = http_serve_file(g_current_root, req->path, ...);
        if (file_len > 0) → 200 OK + 正确的 Content-Type
    }
  该处理器位于 /blog 路由之后、/hello 等 API 路由之前，确保：
  - /blog/* 优先被 blog 专用路由处理（保持向后兼容）
  - 其他任意路径从当前虚拟主机的 root 提供

验证结果：
  curl -H "Host: blog.localhost" http://127.0.0.1:8080/css/style.css
  → HTTP/1.1 200 OK, Content-Type: text/css; charset=utf-8  ✓

  curl -H "Host: blog.localhost" http://127.0.0.1:8080/js/main.js
  → HTTP/1.1 200 OK, Content-Type: application/javascript  ✓

  curl -H "Host: blog.localhost" http://127.0.0.1:8080/img/logo.png
  → HTTP/1.1 200 OK, Content-Type: image/png  ✓

问题 2：blog.local 域名不解析导致 502
================================================================

问题现象：
  浏览器访问 http://blog.local:8080/ 返回 502 Bad Gateway。

定位过程：
  1. blog.local 不是特殊域名，需要 DNS 解析或 /etc/hosts 条目。
  2. 系统中未配置 blog.local → DNS 查询可能返回一个外部 IP。
  3. 外部 IP 上的某个反向代理返回 502。

原因分析：
  用户习惯使用 blog.localhost（*.localhost 在所有系统上自动指向
  127.0.0.1），但配置中只写了 blog.local。blog.local 在无 DNS 配置
  的环境下无法解析到本机。

修改内容：
  conf/server.conf: 每个 server block 的 server_name 同时添加
  *.localhost 和 *.local 两种形式，开箱即用：
    server_name blog.local blog.localhost
    server_name lab.local lab.localhost

问题 3：端口 80 需要 root 权限导致 bind 失败
================================================================

问题现象：
  ./mini_web_server master conf/server.conf
  → [ERROR] bind(0.0.0.0:80) failed — port may be in use
  → 退出码 255

定位过程：
  1. ss -tlnp 确认端口 80 未被占用。
  2. 查阅 Linux 文档：1024 以下的端口是特权端口，需要 CAP_NET_BIND_SERVICE
     能力或 root 用户才能绑定。

原因分析：
  配置文件中 listen 端口写的是 80（HTTP 默认端口），但普通用户无权绑定
  特权端口。之前的测试都在 8080/8888/8890 等非特权端口上运行。

修改内容：
  conf/server.conf: 所有 listen 端口从 80 改为 8080。

v1.2.1 虚拟主机调试总结
================================================================

| 问题 | 根因 | 修复位置 |
|------|------|---------|
| CSS/JS/图片 404 | 服务器只有 / 和 /blog 两个静态路由，无通用文件处理器 | request_handler.c (新增通用静态文件路由) |
| blog.local 返回 502 | 域名无 DNS 解析，请求发到了外部 IP | server.conf (添加 *.localhost 域名) |
| 端口 80 bind 失败 | 普通用户无权绑定特权端口 | server.conf (80→8080) |

经验教训：
1. 静态文件服务器必须支持从根目录提供任意路径的文件，不能只覆盖特定的
   URL 前缀。HTML 中的相对路径引用（css/style.css）不会带 /blog/ 前缀。
2. *.localhost 域名在所有系统上自动指向 127.0.0.1，是测试虚拟主机的
   最佳选择——无需编辑 /etc/hosts。
3. 默认配置文件应使用 >1024 的非特权端口，避免用户首次启动就遇到 bind
   权限错误。
4. 名称虚拟主机（name-based virtual hosting）的核心链路只有两步：
   a) 从请求中提取 Host 头
   b) 将 Host 与配置中的 server_name 做大小写不敏感精确匹配
   匹配后选中对应的文档根目录（root），后续处理与单网站模式完全相同。
